# UI设计规范文档

文档版本：V1.0

更新日期：2026年1月29日

适用范围：产品所有UI设计、原型设计、前端开发工作，覆盖移动端（iOS/Android）、PC端Web页面及小程序端，确保全平台设计风格统一、交互一致，提升设计效率、降低协作成本，为用户提供连贯流畅的使用体验。

## 一、总则

### 1.1 规范目的

1.  统一产品视觉风格，强化品牌辨识度，避免设计碎片化；

2.  简化设计流程，减少重复设计工作，提升团队协作效率（设计→开发→测试）；

3.  规范交互逻辑，降低用户学习成本，提升产品易用性和一致性；

4.  为前端开发提供明确依据，减少还原偏差，降低沟通和迭代成本；

5.  便于后续产品迭代和维护，确保新增功能与现有设计风格统一。

### 1.2 核心原则

- 一致性：视觉风格、交互逻辑、组件使用、文案规范全平台统一；

- 易用性：优先满足用户核心需求，设计简洁直观，避免复杂操作和冗余元素；

- 美观性：符合现代设计审美，兼顾视觉层次感和细节质感，贴合品牌调性；

- 可扩展性：规范预留迭代空间，支持新增组件、适配新场景，避免过度僵化；

- 可实现性：设计需结合前端技术能力，避免无法落地的设计方案。

### 1.3 适用人群

UI设计师、UX设计师、产品经理、前端开发工程师、测试工程师、运营设计师。

## 二、基础设计规范

### 2.1 色彩规范

色彩是品牌识别的核心元素，需严格遵循以下规范，禁止随意使用未定义色彩，确保全平台色彩统一。

#### 2.1.1 品牌色彩

| 色彩类型 | 色值（Hex/RGB） | 用途说明 |

| --- | --- | --- |

| 主色 | #1890FF（RGB:24,144,255） | 品牌核心色，用于按钮、导航栏、重点强调元素、关键操作入口 |

| 主色辅助色 | #096DD9（RGB:9,109,217）、#40A9FF（RGB:64,169,255） | 主色的深浅变体，用于hover状态、渐变效果、层级区分 |

| 辅助色 | 成功：#52C41A（RGB:82,196,26）；警告：#FAAD14（RGB:250,173,20）；错误：#F5222D（RGB:245,34,45）；信息：#8C8C8C（RGB:140,140,140） | 用于状态提示、反馈信息、标签分类（如成功提示、错误警告、信息说明） |

#### 2.1.2 中性色彩

| 色彩名称 | 色值（Hex/RGB） | 用途说明 |

| --- | --- | --- |

| 背景色（一级） | #FFFFFF（RGB:255,255,255） | 页面主背景、卡片背景、组件背景（默认状态） |

| 背景色（二级） | #F5F7FA（RGB:245,247,250） | 页面次要背景、区域分隔背景、列表交替背景 |

| 文本色（一级） | #1D2129（RGB:29,33,41） | 主标题、正文文本、重要提示文本 |

| 文本色（二级） | #4E5969（RGB:78,89,105） | 副标题、辅助文本、次要说明 |

| 文本色（三级） | #86909C（RGB:134,144,156） | 占位文本、提示文本、底部说明文本 |

| 边框色 | #E5E6EB（RGB:229,230,235） | 组件边框、卡片边框、区域分隔线 |

#### 2.1.3 色彩使用禁忌

1.  禁止将主色与高饱和对比色（如红色与绿色）直接搭配，避免视觉刺眼；

2.  文本与背景色对比度需≥4.5:1（重要文本），确保可读性，禁止浅色文本搭配浅色背景；

3.  同一页面内，色彩种类不超过4种（品牌色+1-2种辅助色+中性色），避免色彩杂乱；

4.  辅助色（成功/警告/错误）仅用于对应状态提示，不用于常规装饰。

### 2.2 字体规范

字体统一采用系统默认字体，兼顾跨平台兼容性和可读性，明确字体层级，避免混乱。

#### 2.2.1 字体家族

- PC端Web：优先使用「Microsoft YaHei」（微软雅黑），备选「PingFang SC」（苹方）、「Helvetica Neue」；

- 移动端（iOS）：优先使用「PingFang SC」（苹方）；

- 移动端（Android）：优先使用「Roboto」，备选「PingFang SC」；

- 小程序端：统一使用「PingFang SC」（苹方），确保与微信生态适配。

#### 2.2.2 字体层级

| 层级 | 字号（px） | 字重 | 用途说明 |

| --- | --- | --- | --- |

| 一级标题 | 24 | 600（半粗） | 页面主标题、弹窗标题、重要模块标题 |

| 二级标题 | 20 | 600 | 模块标题、卡片标题、次级页面标题 |

| 三级标题 | 18 | 500（常规粗） | 列表标题、小组件标题、辅助模块标题 |

| 正文（一级） | 16 | 400（常规） | 核心正文、操作提示、按钮文本 |

| 正文（二级） | 14 | 400 | 辅助正文、说明文本、列表内容 |

| 辅助文本 | 12 | 400 | 占位文本、标签文本、底部说明、时间戳 |

#### 2.2.3 字体使用规则

1.  字号严格遵循上述层级，禁止随意调整（特殊场景需经设计负责人确认）；

2.  字重仅使用400（常规）、500（常规粗）、600（半粗），禁止使用700及以上加粗（品牌Logo除外）；

3.  文本对齐：标题左对齐（特殊居中场景除外），正文左对齐，数字、时间戳右对齐；

4.  行高：标题行高1.2-1.3，正文行高1.5-1.6，辅助文本行高1.4，确保可读性；

5.  禁止使用斜体字体，下划线仅用于链接（颜色统一为主色#1890FF）。

### 2.3 间距与栅格规范

统一间距和栅格，确保页面布局整齐有序，元素对齐一致，提升视觉协调性。

#### 2.3.1 基础间距

基础间距采用8px为基准单位，遵循“2倍原则”，常用间距为：4px、8px、16px、24px、32px、48px，禁止使用非标准间距（如5px、7px、15px）。

| 间距尺寸（px） | 用途说明 |

| --- | --- |

| 4 | 微小间距，用于组件内部元素间距（如图标与文本、按钮内部间距） |

| 8 | 基础间距，用于组件之间、列表项内部、文本与边框的间距 |

| 16 | 常用间距，用于模块之间、页面内边距（左右）、卡片内部间距 |

| 24 | 较大间距，用于页面上下内边距、重要模块与其他模块的分隔 |

| 32/48 | 超大间距，用于页面顶部标题与内容的分隔、全局模块的分隔（少见） |

#### 2.3.2 栅格系统

PC端Web页面采用12列栅格系统，移动端采用6列栅格系统，小程序端采用8列栅格系统，栅格间距统一为16px，左右边距统一为16px（PC端可根据屏幕尺寸调整为24px）。

1.  PC端：屏幕宽度≥1200px时，栅格总宽度1152px；屏幕宽度768-1199px时，栅格总宽度自适应；

2.  移动端：屏幕宽度≤767px时，栅格总宽度自适应，列宽平均分配；

3.  栅格使用规则：组件、模块需贴合栅格列数，禁止跨列错位布局；重要内容优先占据左侧栅格。

### 2.4 圆角与阴影规范

#### 2.4.1 圆角规范

圆角尺寸统一为4px、8px、16px，根据组件尺寸和场景选择，避免随意设置圆角。

- 4px：小型组件（按钮、输入框、标签、图标）；

- 8px：中型组件（卡片、弹窗、列表项、面板）；

- 16px：大型组件（模态框、轮播图、特殊模块容器）；

- 特殊说明：圆形元素（头像、图标）采用50%圆角，禁止混合使用不同圆角尺寸的同类型组件。

#### 2.4.2 阴影规范

阴影用于区分元素层级，增强视觉立体感，仅在需要突出的组件上使用，禁止过度使用阴影导致页面杂乱。

| 阴影类型 | 阴影参数 | 用途说明 |

| --- | --- | --- |

| 基础阴影 | box-shadow: 0 2px 8px 0 rgba(0,0,0,0.08) | 卡片、列表项、面板（默认状态） |

| hover阴影 | box-shadow: 0 4px 16px 0 rgba(0,0,0,0.12) | 组件hover状态（卡片、按钮、列表项） |

| 突出阴影 | box-shadow: 0 8px 24px 0 rgba(0,0,0,0.16) | 弹窗、模态框、当前激活的组件 |

### 2.5 图标规范

图标用于辅助文本，增强视觉识别性，统一图标风格、尺寸和用法，确保全平台一致性。

#### 2.5.1 图标风格

统一采用「线性图标」，线条粗细统一为2px，风格简洁、扁平化，避免使用写实、渐变、立体图标；图标边角采用圆角处理（2px），避免尖锐边角。

#### 2.5.2 图标尺寸

常用图标尺寸为16px、24px、32px，根据组件大小匹配，禁止使用非标准尺寸。

- 16px：辅助文本图标、列表项图标、小型按钮图标；

- 24px：常规按钮图标、导航栏图标、模块标题图标；

- 32px：重点图标、空状态图标、引导页图标。

#### 2.5.3 图标用法

1.  图标颜色：默认使用文本色二级（#4E5969），激活/选中状态使用主色（#1890FF），禁用状态使用文本色三级（#86909C）；

2.  图标与文本搭配：图标在左，文本在右，间距统一为8px；

3.  图标一致性：相同功能的图标必须统一（如“编辑”图标仅使用一种样式），禁止混用；

4.  空状态图标：需搭配简洁文本提示，图标与文本间距为16px，居中显示。

## 三、组件设计规范

组件是UI设计的基础单元，所有组件需遵循统一样式和交互逻辑，确保复用性和一致性，以下为核心常用组件规范。

### 3.1 按钮组件

#### 3.1.1 按钮类型与样式

| 按钮类型 | 样式参数 | 用途说明 |

| --- | --- | --- |

| 主按钮 | 背景色：#1890FF；文本色：#FFFFFF；圆角：4px；hover背景色：#096DD9；禁用背景色：#B3D8FF | 核心操作、确认操作（如提交、保存、确认） |

| 次按钮 | 背景色：#FFFFFF；文本色：#1890FF；边框：1px solid #1890FF；圆角：4px；hover背景色：#F5F7FA；禁用文本色：#86909C | 次要操作、辅助操作（如取消、返回、重置） |

| 文本按钮 | 背景色：透明；文本色：#1890FF；无边框；圆角：4px；hover背景色：#F5F7FA；禁用文本色：#86909C | 轻微操作、辅助操作（如查看详情、删除、编辑） |

| 状态按钮 | 成功：背景色#52C41A；警告：背景色#FAAD14；错误：背景色#F5222D；文本色均为#FFFFFF；圆角4px | 对应状态的操作（如确认成功、警告提示、删除确认） |

#### 3.1.2 按钮尺寸

常用按钮尺寸分为三种，根据页面布局和操作优先级选择。

- 大型按钮：高度48px，字号16px，内边距左右24px（用于页面核心操作，如顶部提交按钮）；

- 常规按钮：高度40px，字号16px，内边距左右16px（默认使用，如表单提交、列表操作）；

- 小型按钮：高度32px，字号14px，内边距左右8px（用于紧凑布局、辅助操作，如标签内按钮）。

#### 3.1.3 按钮交互规范

1.  hover状态：所有按钮hover时需有明显反馈（背景色/边框色变化），无延迟；

2.  点击状态：点击时按钮轻微下沉（或背景色加深），反馈时间≤100ms；

3.  禁用状态：禁用时按钮不可点击，样式灰度化，鼠标悬浮显示“禁止”图标；

4.  按钮排列：多个按钮并列时，主按钮在左，次按钮/文本按钮在右，间距统一为16px；

5.  按钮文本：文本简洁明了，不超过8个字（如“提交”“保存”“取消”），禁止换行。

### 3.2 输入框组件

#### 3.2.1 输入框样式

- 常规输入框：高度40px，边框1px solid #E5E6EB，圆角4px，背景色#FFFFFF；

- 聚焦状态：边框颜色变为#1890FF，添加轻微阴影（box-shadow: 0 0 0 2px rgba(24,144,255,0.1)）；

- 禁用状态：背景色#F5F7FA，边框颜色#E5E6EB，文本色#86909C，不可输入；

- 错误状态：边框颜色#F5222D，背景色#FFFFFF，下方显示错误提示文本（字号12px，颜色#F5222D，间距4px）；

- 占位文本：颜色#86909C，字号14px，文本左对齐，内边距左右16px。

#### 3.2.2 输入框类型

- 单行输入框：默认类型，用于输入姓名、手机号、密码等短文本；

- 多行输入框：高度自适应（默认高度60px，最大高度120px），用于输入备注、描述等长文本，可换行；

- 带图标准输入框：图标在左（尺寸16px，颜色#86909C），与输入文本间距8px，用于区分输入类型（如手机号、邮箱）；

- 带清除按钮输入框：输入内容后，右侧显示清除图标（16px），点击清除输入内容。

#### 3.2.3 输入框交互规范

1.  聚焦：点击输入框或Tab键切换时，自动聚焦，光标颜色#1890FF，光标高度为字号的80%；

2.  输入限制：根据输入类型设置最大长度（如手机号11位），超出长度禁止输入，可显示输入计数（右下角，字号12px，颜色#86909C）；

3.  清除按钮：仅当有输入内容时显示，hover时图标颜色变为#1890FF；

4.  错误提示：输入内容不符合规则时，实时显示错误提示（或失去焦点后显示），修正后错误提示自动消失；

5.  多行输入：输入超出默认高度时，自动换行并增高，最大高度后出现滚动条，禁止横向滚动。

### 3.3 卡片组件

#### 3.3.1 卡片样式

- 常规卡片：背景色#FFFFFF，边框1px solid #E5E6EB，圆角8px，阴影为基础阴影（box-shadow: 0 2px 8px 0 rgba(0,0,0,0.08)）；

- hover状态：阴影变为hover阴影（box-shadow: 0 4px 16px 0 rgba(0,0,0,0.12)），无其他样式变化；

- 卡片内部间距：上下24px，左右16px（常规卡片）；上下16px，左右16px（小型卡片）；

- 卡片标题：字号18px，字重500，颜色#1D2129，下方可添加分割线（1px solid #E5E6EB，间距8px）。

#### 3.3.2 卡片类型与用途

- 数据卡片：用于展示核心数据（如数值、图表），标题下方为数据内容，可搭配图标辅助展示；

- 列表卡片：用于展示列表项（如订单、用户信息），内部包含多个元素（文本、图标、按钮），可添加点击事件；

- 详情卡片：用于展示详细信息（如用户详情、商品详情），内部元素按网格或线性排列，层级清晰；

- 空白卡片：用于空状态展示（如无数据、无订单），包含图标（32px）、文本提示（14px，颜色#86909C），居中显示。

#### 3.3.3 卡片交互规范

1.  可点击卡片：点击卡片任意区域（除内部按钮外）触发跳转或展开详情，hover时显示指针图标（cursor: pointer）；

2.  卡片排列：多个卡片并列时，间距统一为16px，对齐方式为上对齐，尺寸一致；

3.  卡片折叠：可折叠卡片右侧添加折叠图标（16px），展开时图标旋转180°，折叠时隐藏内部内容；

4.  卡片选中：需要选中的卡片（如订单选择），选中后边框颜色变为#1890FF，添加轻微边框加粗（2px）。

### 3.4 弹窗组件

#### 3.4.1 弹窗样式

- 弹窗容器：背景色#FFFFFF，圆角8px，阴影为突出阴影（box-shadow: 0 8px 24px 0 rgba(0,0,0,0.16)），宽度：PC端480px，移动端自适应（左右边距16px）；

- 弹窗标题栏：高度48px，边框底部1px solid #E5E6EB，标题字号18px，字重500，颜色#1D2129，左对齐，内边距左右16px；

- 弹窗内容区：内边距24px，文本字号14px，颜色#4E5969，行高1.5，可包含输入框、按钮、列表等组件；

- 弹窗按钮区：内边距16px，边框顶部1px solid #E5E6EB，按钮排列在右侧，间距16px；

- 遮罩层：背景色rgba(0,0,0,0.5)，点击遮罩层可关闭弹窗（特殊场景除外）。

#### 3.4.2 弹窗类型与用途

- 确认弹窗：用于确认重要操作（如删除、提交），内容区为提示文本，按钮区为“确认”（主按钮）和“取消”（次按钮）；

- 输入弹窗：用于快速输入内容（如备注、名称），内容区包含输入框，按钮区为“确认”和“取消”；

- 详情弹窗：用于展示详细信息（如订单详情、日志详情），内容区为详情内容，按钮区为“关闭”（次按钮）；

- 提示弹窗：用于展示提示信息（如操作成功、失败），无标题栏，内容区包含图标和文本，自动关闭（默认3秒）或点击关闭。

#### 3.4.3 弹窗交互规范

1.  弹窗出现：从页面中心弹出，伴有轻微淡入动画（时长150ms），遮罩层同步淡入；

2.  弹窗关闭：点击关闭按钮、遮罩层、取消按钮时，弹窗淡出消失（时长150ms），遮罩层同步淡出；

3.  输入弹窗：输入内容不符合规则时，禁止点击确认按钮，显示错误提示；

4.  提示弹窗：成功提示用绿色图标，失败提示用红色图标，信息提示用灰色图标，文本与图标间距8px；

5.  弹窗层级：弹窗层级高于页面所有组件，遮罩层层级低于弹窗，禁止多个弹窗同时显示。

## 四、页面设计规范

页面设计需遵循基础规范和组件规范，确保页面布局整齐、层级清晰、交互流畅，全平台页面风格统一。

### 4.1 页面结构规范

所有页面统一采用“顶部导航栏 + 侧边栏（PC端） + 内容区 + 底部版权区”的结构（特殊页面除外，如引导页、登录页）。

#### 4.1.1 顶部导航栏

- 高度：PC端64px，移动端56px；

- 背景色：#FFFFFF，PC端添加基础阴影，移动端无阴影；

- 内容：左侧为品牌Logo（尺寸：PC端40px，移动端32px），中间为导航菜单（PC端），右侧为用户头像、通知图标、退出按钮；

- 交互：PC端导航菜单hover时显示下拉菜单，移动端导航菜单点击显示侧边导航栏。

#### 4.1.2 侧边栏（PC端）

- 宽度：200px（展开）、80px（折叠）；

- 背景色：#FFFFFF，边框右侧1px solid #E5E6EB；

- 内容：导航菜单（图标+文本，图标24px，文本14px，间距8px），当前激活菜单背景色#E6F7FF，文本色#1890FF；

- 交互：点击菜单可展开/折叠子菜单，hover菜单时背景色#F5F7FA，支持折叠/展开切换。

#### 4.1.3 内容区

- 背景色：#F5F7FA（PC端）、#FFFFFF（移动端）；

- 内边距：PC端上下24px、左右24px；移动端上下16px、左右16px；

- 内容布局：遵循栅格系统，模块之间间距16px-24px，层级清晰（标题→筛选→列表/卡片→分页）；

- 空状态：无数据时，显示空状态卡片（图标32px，文本14px，颜色#86909C），居中显示。

#### 4.1.4 底部版权区

- 高度：48px；

- 背景色：#FFFFFF（PC端）、#F5F7FA（移动端）；

- 内容：版权信息（字号12px，颜色#86909C），居中显示，可添加联系方式、隐私政策链接（文本色#1890FF）。

### 4.2 页面适配规范

支持响应式适配，确保在不同设备尺寸下页面正常显示，交互流畅，核心内容不缺失。

#### 4.2.1 适配尺寸

- PC端：屏幕宽度≥1200px（大屏）、768px-1199px（中屏）、≤767px（小屏，适配移动端）；

- 移动端：屏幕宽度320px-414px（手机）、415px-767px（平板）；

- 小程序端：适配微信小程序标准尺寸，遵循微信小程序设计规范。

#### 4.2.2 适配规则

1.  PC端小屏适配：侧边栏折叠，导航菜单简化，卡片排列从多列变为2列；

2.  移动端适配：隐藏侧边栏，顶部导航栏简化（仅保留Logo、菜单按钮、用户图标），卡片排列变为1列，按钮尺寸适配移动端（常规按钮高度36px）；

3.  图片适配：图片宽度自适应，高度按比例缩放，禁止图片拉伸、变形；

4.  文本适配：移动端文本字号可适当缩小（如正文从16px变为14px），确保可读性；

5.  交互适配：移动端取消hover交互，改为点击交互；添加手势支持（如下拉刷新、左右滑动删除）。

### 4.3 常见页面规范

#### 4.3.1 登录页

- 布局：居中布局（表单容器居中，背景色#FFFFFF，圆角8px，阴影突出）；

- 内容：品牌Logo（居中，尺寸48px）、登录表单（输入框2个：账号/手机号、密码，按钮1个：登录，文本链接：忘记密码、注册）；

- 样式：输入框高度40px，登录按钮高度48px（主按钮），表单内间距16px，Logo与表单间距24px；

- 交互：密码输入框可切换显示/隐藏，登录按钮禁用状态（未输入内容时），登录失败显示错误提示（文本12px，颜色#F5222D）。

#### 4.3.2 列表页

- 布局：顶部筛选区（搜索框、下拉选择、筛选按钮），中间列表区，底部分页区；

- 筛选区：搜索框（带图标准输入框），下拉选择（与输入框并排，间距16px），筛选按钮（主按钮），重置按钮（文本按钮）；

- 列表区：采用卡片列表或表格列表，列表项高度48px，hover背景色#F5F7FA，选中项背景色#E6F7FF；

- 分页区：居中显示，页码间距8px，当前页码样式（背景色#1890FF，文本色#FFFFFF，圆角4px），支持上一页、下一页、跳转页码。

#### 4.3.3 详情页

- 布局：顶部标题区（标题、返回按钮、编辑按钮），中间详情区（卡片布局，分模块展示），底部操作区（相关按钮）；

- 标题区：标题字号20px，返回按钮（文本按钮，带返回图标），编辑按钮（文本按钮，带编辑图标），间距16px；

- 详情区：采用卡片分模块展示，每个卡片包含标题（18px）和内容，卡片间距16px；

- 操作区：按钮排列在右侧，间距16px，核心操作（如保存、提交）为主按钮，次要操作为次按钮/文本按钮。

## 五、交互设计规范

交互设计需遵循“简洁、流畅、直观”的原则，统一交互逻辑和反馈机制，提升用户体验，减少用户学习成本。

### 5.1 反馈机制规范

所有用户操作都需有明确反馈，反馈及时、清晰，不干扰用户正常操作。

#### 5.1.1 操作反馈

- 点击反馈：按钮、卡片、列表项等可点击元素，点击时需有状态变化（如背景色加深、边框变色），反馈时间≤100ms；

- 加载反馈：页面加载、数据请求时，显示加载动画（如Spinner加载图标，尺寸24px，颜色#1890FF），居中显示；加载失败时，显示失败提示（图标+文本，可点击重试）；

- 成功/失败反馈：操作成功（如保存、提交）显示绿色提示（图标+文本，3秒后自动消失）；操作失败显示红色提示（图标+文本，可点击关闭或重试）；

- 输入反馈：输入内容符合规则时，输入框右侧显示成功图标（16px，颜色#52C41A）；不符合规则时，显示错误图标和提示文本。

#### 5.1.2 反馈样式统一

- 提示文本：字号14px，行高1.5，与图标间距8px；

- 提示位置：页面顶部居中（全局提示）、组件下方（局部提示，如输入框错误提示）、弹窗内（弹窗提示）；

- 加载动画：统一使用Spinner图标，禁止使用自定义加载动画；

- 反馈时长：自动关闭的提示（成功/失败），时长3秒；加载动画持续至操作完成。

### 5.2 导航交互规范

#### 5.2.1 顶部导航

- PC端：导航菜单hover时显示下拉菜单（淡入动画，时长150ms），下拉菜单背景色#FFFFFF，圆角4px，阴影基础阴影；点击菜单跳转至对应页面，当前页面菜单高亮；

- 移动端：点击导航菜单按钮，显示侧边导航栏（从左侧滑入，时长150ms），侧边导航栏可点击关闭按钮或遮罩层关闭。

#### 5.2.2 侧边导航（PC端）

- 菜单点击：点击父菜单展开/折叠子菜单（滑入/滑出动画，时长150ms），子菜单缩进16px，图标尺寸16px；

- 高亮状态：当前页面菜单背景色#E6F7FF，文本色#1890FF，图标颜色#1890FF；

- 折叠/展开：点击折叠按钮，侧边栏从200px缩至80px（仅显示图标），再次点击展开，动画时长200ms。

#### 5.2.3 页面跳转

- 跳转方式：内部页面跳转（如列表页→详情页），无页面刷新动画；外部页面跳转，新窗口打开；

- 返回逻辑：详情页→列表页、子页面→父页面，点击返回按钮可返回上一级页面，保留上一级页面的筛选条件和分页状态；

- 面包屑导航（PC端）：页面顶部添加面包屑（如“首页→用户管理→用户列表”），文本色#4E5969，当前页面文本色#1890FF，分隔符为“/”，点击可跳转至对应页面。

### 5.3 表单交互规范

- 输入逻辑：输入框聚焦时自动选中原有内容（便于修改），支持Tab键切换输入框（按从左到右、从上到下的顺序）；

- 验证逻辑：实时验证输入内容，不符合规则时立即反馈（如手机号格式错误、密码长度不足），验证规则与输入框错误状态一致；

- 提交逻辑：表单填写完成且验证通过后，提交按钮才可点击；提交过程中显示加载状态，禁止重复点击；提交成功/失败后显示对应反馈，成功后可跳转至目标页面或重置表单；

- 重置逻辑：点击重置按钮，清空所有输入内容，恢复表单默认状态（如占位文本、未选中状态），重置后无额外反馈；

- 表单布局：标签与输入框上下排列（移动端）或左右排列（PC端），标签文本色#1D2129，字号14px，标签与输入框间距8px；多个输入框之间间距16px，表单与按钮间距24px。

## 六、文案规范

文案是UI设计的重要组成部分，统一文案风格和规范，确保文案简洁、准确、易懂，贴合产品调性，提升用户体验。

### 6.1 文案风格

- 简洁明了：避免冗长、复杂的表述，用最简洁的语言传递核心信息，禁止冗余修饰；

- 准确规范：用词准确，无歧义，符合中文语法规范，禁止错别字、病句；

- 友好亲切：语气温和、亲切，避免生硬、命令式表述，贴合产品目标用户群体；

- 统一调性：全平台文案风格一致（如正式、简约、活泼），禁止不同页面文案风格差异过大。

### 6.2 具体文案规范

#### 6.2.1 按钮文案

- 简洁：不超过8个字，优先使用动词+名词或纯动词结构（如“提交表单”“保存”“删除”）；

- 明确：文案需清晰表达按钮功能，禁止模糊表述（如禁止使用“确定”代替“提交”“保存”）；

- 统一：相同功能的按钮文案必须统一（如“取消”按钮不可有时用“取消”，有时用“返回”）。

#### 6.2.2 提示文案

- 错误提示：明确指出错误原因，给出解决方案（如“手机号格式错误，请输入11位纯数字”），禁止模糊提示（如“输入错误”）；

- 成功提示：简洁表达操作结果，可搭配后续指引（如“保存成功，将自动返回列表页”）；

- 占位提示：提示用户需输入的内容，简洁明了（如“请输入手机号”“请输入备注信息”），禁止使用“请输入内容”等模糊提示；

- 警告提示：明确指出风险或问题，引导用户正确操作（如“此操作将删除数据，不可恢复，请确认”）。

#### 6.2.3 页面文案

- 标题文案：简洁有力，概括页面核心内容，不超过20个字，一级标题加粗，贴合层级规范；

- 正文文案：条理清晰，段落分明，行高符合字体规范，避免大段无分隔的文本；

- 链接文案：明确链接指向，简洁明了（如“查看详情”“前往设置”），颜色统一为主色#1890FF，添加下划线。

### 6.3 文案禁忌

1.  禁止使用错别字、病句、网络用语（如“yyds”“绝绝子”）、口语化过重的表述；

2.  禁止使用模糊、歧义的表述，避免用户误解；

3.  禁止使用命令式语气（如“必须输入”“立即点击”），改为引导式语气（如“请输入”“建议点击”）；

4.  禁止文案冗长，避免大段文本堆积，必要时拆分段落或使用列表；

5.  全平台文案统一，禁止同一功能、同一提示有不同表述。

## 七、规范维护与更新

为确保规范的时效性和适用性，需建立规范维护机制，及时更新迭代，适配产品发展需求。

### 7.1 维护责任

- 设计负责人：负责规范的整体维护、更新审核，确保规范的统一性和权威性；

- UI设计师：负责收集规范使用过程中的问题，提出更新建议，协助完善规范；

- 前端开发工程师：负责反馈规范在开发落地过程中的可实现性问题，提供优化建议。

2.  规范更新前，需征求设计、开发、产品等相关团队意见，确保更新内容合理、可落地；
